## 1. 背景与目标

`Auto_review` 现在是一套 **Windows + PowerShell 下的 Python 命令行流水线**:把论文 PDF 变成
可溯源、可重组的综述写作素材,再由 AI 接地出稿。入口是 `interactive_assistant.py` 的菜单和
`.bat` 启动器,数据以 JSON 散落在本地 `library/`、`reports/`。

目标是把它做成一个**用户可视化的工具**,分三阶段推进:

1. **桌面试用版(本文件)**:自己用 + 放给论坛用户体验,收集"这工具有没有用"的反馈。
2. **云端正式版(将来)**:把后端搬上服务器,支持多用户。
3. **手机端(将来)**:复用同一套界面与后端。

核心设计原则贯穿三阶段:**引擎不重写,只换一张脸**。现有 Python 流水线是真正的资产,
本阶段在它前面加一层本地接口 + 一张网页界面;同一套后端将来原样搬到服务器即进入第二阶段。

### 本阶段的运行定位(做法 A:本地分发)

- 桌面程序装在**用户自己电脑**上(Windows / macOS)。
- 用户**自带 API key**(DeepSeek),**自带论文 PDF**。
- AI 费用、PDF 版权与合规风险**全部留在用户本地**,开发者不架服务器、不代存任何用户数据。

## 2. 范围与非目标

### 范围内(v1)

- 桌面应用:Windows + macOS。
- **发现 + 下载层(插件框架)**:软件内搜索开放获取文献、导入题录(RIS/BibTeX 等)、勾选后统一下载;下载来源做成可扩展插件(OA 自带;Sci-Hub / 截图下载为可选插件)。
- 完整流水线:搜索/导入题录 → 下载 PDF → 抽取 → 关联网 → 找灵感 → 追问式写综述 → 出稿+溯源。
- **单篇拆解视图**:点开论文看结构化拆解(摘要要点/问题/术语表/分析-结果/结果关联),全程点击溯源;依赖精细层回归 + 新增术语表抽取。
- **课题组归类**:按资深作者(PI)身份为主、合著社群为辅,把同组文章自动归堆。
- **可插拔 PDF 抽取后端**:PyMuPDF(默认,轻) / Docling(可选,重,按需安装)。
- 用户自带 API key(存系统钥匙串)。

### 非目标(本阶段明确不做)

- 不做云服务、不做账号/登录/权限。
- 不做手机端。
- 不预置任何"样板库";空库启动,由用户自行搜索/导入。
- 下载层不绕过任何登录、付费墙或访问控制;Sci-Hub 插件默认关闭且不内置镜像地址。
- 本阶段内置搜索仅对接开放获取/元数据接口(OpenAlex/Crossref/arXiv),不做通用网页爬虫。
- 不代存用户 PDF、不上传用户数据到开发者服务器。
- 不重写现有 Python 引擎(只加薄层包装)。

## 3. 总体架构

四层,从里到外:

```text
[发动机] 现有 Python 流水线(不改)
   docdecomp 包 + scripts/ 抽取与连接层脚本 + run_workflow_with_recovery 等

[插头]   FastAPI 本地接口层(新增,Python)
   把"跑某一步 / 查某份数据 / 起一个长任务"暴露成 localhost 上的 HTTP 接口

[索引]   SQLite + 文件夹(新增)
   SQLite = 引擎 JSON 产物的只读索引/缓存,供界面快速查询;PDF 与导出件存文件夹

[脸]     网页前端(HTML/JS),由桌面外壳(pywebview)包成原生窗口
```

此外,**发现 + 下载层**(软件内搜索台 + 可扩展下载插件)挂在引擎之前,负责把文献从
网络/题录变成本地 PDF,再喂给流水线;它本身也通过 FastAPI 接口对前端暴露,详见第 5A 节。

**关键架构性质**:界面(网页)与后端(FastAPI)之间只通过 HTTP 接口通信。本阶段
后端跑在 localhost;第二阶段把同一个 FastAPI 服务搬到服务器,前端几乎不改即可联网——
这就是"一次搭骨架,扛完三阶段"。

### 桌面外壳选型

- **默认:pywebview**。纯 Python,调用系统自带 webview 把前端网页包成原生窗口,
  与 Python 后端同进程/同打包链路最简单,适合单人开发的试用阶段。
- **可选升级:Tauri**(Rust 外壳,二进制更小、更精致),作为产品化后的潜在替换。
  因前端是标准网页技术,换外壳不影响界面代码。

> 选型说明:此处选 pywebview 是为减少打包复杂度(无需 Rust 工具链);前端保持
> 网页技术以保留将来换壳与上浏览器/手机的自由。最终确认放在实现计划阶段。

## 4. 运行模式

- **启动**:用户双击图标 → 进程内拉起本地 FastAPI 服务(绑定 `127.0.0.1` 随机端口,仅本机可达)
  → 弹出 pywebview 窗口加载前端。用户感知上就是一个普通桌面软件,不暴露"服务器"概念。
- **长任务异步化**:抽取、出稿等几十分钟级重活,不走同步请求,改为**后台任务 + 实时进度**。
  直接复用现有"带恢复的全库运行"思路(`run_workflow_with_recovery.py`),失败可续跑。
- **存储**:
  - **真理源仍是引擎产物**:引擎照旧把卡片/关联写成 `library/Sxx/*.json` 等 JSON(符合
    项目"信产物不信描述"的元原则)。
  - **SQLite 只是索引**:一个进程读引擎 JSON,建立可查询的 SQLite 索引,供界面秒级浏览/搜索。
    SQLite 损坏可随时从 JSON 重建。
  - PDF 原件、导出的稿件存本地文件夹。
- **密钥**:用户的 API key 存**操作系统钥匙串**(Windows Credential Manager / macOS Keychain),
  不写明文配置、不进任何日志、绝不打进安装包。
- **安装与依赖(一次装齐,可靠优先)**:所有**轻量**依赖(Python 运行时、主程序、FastAPI、
  pywebview、PyMuPDF、OA 源所需库等)在**安装时一次性装好**;安装前用一张**同意表单**列清"将安装什么",
  用户同意才装、拒绝则不可用(可接受)。设计以"装好后一定可用"为准,**运行时不反复判断"装没装"、
  不假定缺失**。只有**重**的东西(Docling 数 GB)与**可选插件**(Sci-Hub / 截图下载)才走"按需、单独安装"。

## 5. 组件分解

每个组件给出:职责 / 对外接口 / 依赖。

### 5.1 引擎(现有,不改)

- **职责**:PDF 清洗、阅读块、文献卡片、关联网、找灵感、接地出稿与质量门禁。
- **接口**:现有脚本与 `docdecomp` 包的函数/CLI。FastAPI 层调用它们,不修改其内部。
- **依赖**:DeepSeek 云、本地 `ai_cache`、抽取后端(见 5.2)。

### 5.2 PDF 抽取后端(可插拔,新增接口 + 适配两实现)

- **职责**:把一份 PDF 转成引擎能消费的输入。
- **接口(统一抽象)**:定义一个抽取器接口,约定输入(PDF 路径)→ 输出**一份 Docling 兼容的 JSON**;
  引擎现成的 `build_clean_package` 再把这份 JSON 变成素材包(content_blocks 等)。上层只认接口,不知具体实现。
  > 更正:原写"输出 clean package 的既定结构(由抽取器直接产出素材包)"。读 `package_builder.py` 后确认:
  > 引擎是**从一份 Docling JSON 构建素材包**的,所以抽取器的真实接缝是"PDF → Docling 式 JSON",再复用引擎下游。
  > 这样更贴合"引擎不动"。详见 M2a 计划。
  - 实现 A:**PyMuPDF**——轻量、快、随主程序安装;质量一般。**默认**。
  - 实现 B:**Docling**——重(PyTorch + 模型,数 GB)、慢、质量高;**按需安装**。
- **依赖**:PyMuPDF 实现仅依赖 PyMuPDF;Docling 实现依赖现有 `envs/docling` 运行时。
- **契约风险**:现有流水线是按 Docling 输出调过的;PyMuPDF 输出结构需对齐同一接口,
  质量差异由 5.7 的契约测试守住,差异以 `ISSUES.md` 登记,不在文档里断言"等价"。

### 5.3 FastAPI 接口层(新增)

- **职责**:把引擎能力暴露为 localhost HTTP 接口;管理任务生命周期。
- **接口(示意,实现阶段细化)**:
  - `POST /papers/import`(上传/指向本地 PDF,起抽取任务)
  - `GET /library`、`GET /papers/{id}/card`
  - `GET /papers/{id}/decomposition`(单篇拆解视图数据,各条带溯源指针)
  - `GET /network`(关联网数据)
  - `POST /ideation`(找灵感)
  - `POST /writing/session`、`POST /writing/turn`(追问式写综述)
  - `GET /draft/{id}`、`GET /trace/{citation}`(出稿与溯源)
  - `GET /jobs/{id}`、`GET /jobs/{id}/events`(任务进度,SSE/轮询)
  - `GET /groups`、`GET /groups/{id}`(课题组列表 / 某组的论文·成员·时间线)
  - `POST /settings/apikey`(写钥匙串)、`GET /settings/extractor`(查/切抽取后端)
- **依赖**:引擎、任务系统、存储层、密钥管理。

### 5.4 任务/作业系统(新增,可复用现有恢复逻辑)

- **职责**:把长任务排队、后台执行、上报进度、失败可续跑。
- **接口**:`submit(job) -> job_id`;`status(job_id)`;`events(job_id)`(进度流);`cancel(job_id)`。
- **依赖**:复用 `run_workflow_with_recovery` 的分步/恢复语义;状态落 SQLite。

### 5.5 存储/索引层(新增)

- **职责**:读引擎 JSON 产物 → 建 SQLite 索引;管理 PDF 与导出件的本地文件路径。
- **接口**:`reindex(library_dir)`;按论文/标签/关系查询的读接口。
- **依赖**:引擎产物(`library/index.json`、`library/Sxx/*.json`、`reports/`)。
- **不变量**:SQLite 永远可从 JSON 重建;JSON 与 SQLite 冲突时**以 JSON 为准**。

### 5.6 前端(新增,网页技术)

七个"屏",对应第 6 节数据流:

1. **论文库**:卡片列表(元数据 + 标签 + 粗摘要)。空库时引导用户搜索/导入。
   支持**按课题组分组/筛选**(见 5B):点一个课题组,看它名下的论文、成员、发表时间线。
   **点任一篇 → 进入单篇拆解视图(见 5C)**:摘要要点/问题/术语表/分析-结果/结果关联,点击溯源。
2. **关联网**:可交互网络图(支持/矛盾等有类型的边)。
3. **找灵感**:AI 从库里捞角度/主题/争议点。
4. **写综述(核心)**:追问式交互——AI 抛观点 → 用户追问/修正 → 收敛论点 → 接地出稿。
5. **出稿与溯源**:看稿;点引用跳回来源原文段落;导出。
6. **设置**:填/换 API key;选抽取后端(PyMuPDF/Docling);Docling 按需下载入口。
7. **导入自己的 PDF(核心)**:选文件 → 起抽取任务 → 看进度。
- **接口**:只调 5.3 的 HTTP 接口,不直接碰文件系统/引擎。
- **依赖**:FastAPI 层。

### 5.7 桌面外壳与打包(新增)

- **职责**:pywebview 起窗口、管理后端子进程生命周期、首启初始化(建库目录、查抽取后端)。
- **打包/安装**:安装时**一次性装齐所有轻量依赖**(Python 运行时、主程序、FastAPI、pywebview、
  PyMuPDF、OA 源库),安装前用**同意表单**列明清单;用户拒绝则不安装、不可用。只有 Docling(重)与
  Sci-Hub / 截图下载(可选插件)按需单独安装。运行期一律假定轻量依赖已就位。
- **依赖**:pywebview、PyInstaller(或等价打包器)、各平台代码签名(见第 10 节风险)。

### 5.8 密钥管理(新增)

- **职责**:读写系统钥匙串中的 API key;向引擎按需提供,不落明文、不进日志。
- **接口**:`set_key(value)`;`get_key()`(仅供后端进程内调用)。
- **依赖**:`keyring` 或等价系统钥匙串库。

## 5A. 发现 + 下载层(插件框架,重做法)

把"用户只有一点自己方向的文献"变成"有一个能用的起步库":在软件内搜开放获取文献、
或导入题录拿到 DOI,勾选后统一下载,自动进抽取流水线。这是一个独立子系统,工作量不小,
本阶段按用户决定采用**重做法**(真正可扩展的插件框架)。

### 5A.1 两类能力(插件按需声明)

- **search(query) → 结果列表**(题录 + DOI + 可能的 OA 链接)。
- **fetch(doi/metadata) → PDF**。

一个来源插件声明自己支持哪几种能力:有的只会搜(Crossref),有的搜+取都会(arXiv),
有的只会取(Sci-Hub、截图下载)。

### 5A.2 插件框架(重做法的核心代价与收益)

- **统一接口契约 `SourcePlugin`**:元数据(名称、能力集、是否需用户配置、平台限制、合规等级)、
  `search()`、`fetch()`、健康检查。
- **发现与加载**:从指定插件目录加载;每个插件可单独启用/禁用/安装/卸载。
- **隔离与安全**:插件在受控边界内运行——不直接持有用户 API key、不写任意路径;
  单个插件失败不拖垮主程序。
- **可扩展**:接口与打包规范文档化,将来你或他人可加新来源。这是"重做法"相对"轻做法"
  多付出的设计成本,换来的是长期扩展性。
- **风险控制**:实现阶段**先把接口冻结成 v1**,再写内置/官方插件作为参考实现,避免接口反复改动。

### 5A.3 来源清单(v1)

| 来源 | 能搜 | 能取 | 合规定位 | 形态 |
|---|---|---|---|---|
| OpenAlex / Crossref / arXiv | ✅ | OA 链接可取 | 合法 | 核心自带(官方插件) |
| Sci-Hub | ❌(无 DOI 搜不了) | ✅ | 灰色 | 可选插件,**默认关闭**,镜像由用户自填 |
| 截图下载(浏览器自动化) | ❌ | ✅ | 正当(用户自己已登录、有权限的会话) | 可选插件,**仅 Windows** |

> 截图下载插件由现有 `paper_pool/scripts/paper_downloader.py`(图像模板按钮识别 + 用户本机
> Chrome 会话)**重构而来**,不进核心,只作可选 Windows 插件;它不绕过任何登录/验证码/访问控制。

### 5A.4 下载优先级与去重

- 一篇文献取 PDF 的顺序:**OA → 用户启用的 Sci-Hub / 截图下载兜底**(顺序用户可调)。
- 取不到全文的,只保留元数据并标"无全文",不阻断其余文献。
- 下载后复用 `paper_pool` 的 **RIS 解析、SHA-256 去重、题录归一、`pdf_probe`** 逻辑,再进抽取。

### 5A.5 接口(示意,FastAPI 新增)

- `POST /discovery/search`(关键词 → OA 源搜索)
- `POST /discovery/import`(上传 RIS/BibTeX,解析出 DOI + 元数据)
- `POST /download`(勾选清单 → 起批量下载后台任务)
- `GET /sources`(列出已装插件、能力与健康状态)
- `POST /sources/{id}/config`(配置,如 Sci-Hub 镜像地址)

### 5A.6 依赖与复用

- **复用** `paper_pool`:RIS 解析、SHA-256 去重、题录归一、`pdf_probe`。
- **重构而非复用**:`paper_downloader.py` 的浏览器自动化重构成"截图下载"插件,不进核心。
- 批量下载走第 5.4 节的任务系统(后台 + 进度 + 可续跑)。

## 5B. 课题组归类(脚本打底 + AI 兜疑难)

帮用户把"读文献时散落各处、其实出自同一课题组"的文章自动归到一起。元数据里没有
"课题组"字段,所以先选**代理标志**再聚类:**A 为主——按资深作者(PI)身份;C 辅助——按合著社群**。

### 5B.1 设计原则

- **能确定就别用 AI**:聚类主力是确定、可复现、近零成本的脚本;AI 只判脚本拿不准的少数疑难,
  且判完**缓存 + 留痕**,不每次重算(规避项目已知的 AI 非确定性,见 `ISSUES.md` I12)。
- **每个归并都带证据**:记录"凭哪个信号归到一起"(作者 ID / ORCID / 姓名+单位 / 合著社群),
  便于回溯与人工纠错(符合"信产物"元原则)。

### 5B.2 主路 A:按资深作者(PI)身份聚类(脚本)

- **取 group anchor(课题组锚点)**:默认取**通讯作者**;缺失时退回**末位作者**;字段惯例不同的学科
  此规则**可配置**(如数学按字母序署名,末位≠PI,需特殊处理)。
- **解析锚点身份(按可靠度优先级)**:
  1. **OpenAlex 作者 ID**(走 OpenAlex 来的元数据,已消歧,最强)
  2. **ORCID**
  3. **规范化姓名 + 单位**(最弱,易重名)
- 同一锚点身份的论文 → 一个**候选课题组**。

### 5B.3 辅路 C:合著社群(脚本,辅助校正)

- 用作者共同署名建**合著图**,跑社群划分(先连通块,必要时 Louvain/标签传播)。
- 两个用途:
  - **帮合并**:两个 A 候选组若成员高度重叠(同组、锚点名有变体),合并。
  - **帮发现疑难**:同一锚点姓名却落在**互不相连的社群**→ 疑似重名碰撞 → 交 AI 裁决。

### 5B.4 AI 兜疑难(仅对被标记的边界情况)

- 触发条件:姓名相同但单位/社群冲突等脚本判不准的少数情形。
- 输入给 AI:候选两方的合著者、单位、研究主题等上下文,问"是否同一人/同一组"。
- 结果**经 `ai_cache` 缓存、写进归并证据**,可复现、可审。

### 5B.5 产物与数据模型

- 产出 `groups.json`(**真理源**):每组含锚点身份、成员论文、成员作者、归并证据、置信度。
- 每篇论文带"所属课题组 + 来源信号"。
- 同抽取产物一样,**JSON 为真理源,SQLite 只建索引**供界面按组浏览/筛选。

### 5B.6 接口、复用与定位

- FastAPI 增:`GET /groups`(组列表)、`GET /groups/{id}`(某组的论文/成员/时间线)。
- **复用**:`paper_pool` 的姓名规范化、题录解析;`docdecomp` 的 `ai_client`/`ai_cache`。
- **运行时机**:导入/索引阶段作为确定性后处理跑一次;新增论文增量更新。
- 它是关联网之外的**另一种"镜头"**:关联网看"观点怎么连",课题组看"人/团队怎么聚"。

## 5C. 单篇拆解视图(精细层回归 + 术语表 + 点击溯源)

点开一篇 PDF,弹出/跳到一个**结构化阅读页**,把这篇文章拆给你看:摘要几条、Introduction 提了
什么问题、术语表、用了哪些分析及各自结果、结果之间怎么关联——**每一条都能点回原文那段**。

### 5C.1 主体是"展示已有产物",不是从零抽取

对照 `evidence_synthesis.py` 的类型定义,用户要的大半精细层产物**已经在产**:

| 视图条目 | 来源产物(已有) |
|---|---|
| 用了什么分析 | 证据原子 `method` |
| 每个分析什么结果 | 原子 `result` / `quantitative_result` + 综合 `method_result_link` |
| 结果间关联 | 篇内综合 `paper_syntheses`:`mechanism_result_link` / `variable_effect` / `limitation_scope` 等 |
| Introduction 的问题 | 原子 `background` / `scope` + 章节类型 `section_kind` |
| 摘要要点 | 文献卡片粗摘要 / 摘要章节 + 相关原子 |
| **术语表** | **无 → 新增抽取(`glossary`)** |

### 5C.2 关键前提:精细层回归主线

架构 v2 曾把证据原子 + 篇内综合标为"v1 遗留、非必需",默认抽取可能不产它们。本视图**依赖
它们回归**——即:**App 的单篇抽取流水线要(重新)产出 `evidence_atoms.json` + `paper_syntheses.json`**,
并**新增** `glossary.json` 的抽取与校验(`validate_glossary`)。这与 HANDOFF 所述"精细层只在最后
钻取时用"一致——本视图正是那个"钻取"场景。

### 5C.3 质量风险与对策(如实)

- **风险**:精细层准确度有已知问题(`ISSUES.md` I1 论断正确性;记忆中约 7% 证据原子会失败);
  术语表为新抽取,质量需单独评估。
- **对策**:① **全程点击溯源**——每条结果/关联/术语都带 `reading_block` 指针,点击跳回原文,
  用户可自核;② **展示置信度**——原子已带 `high/medium/low`,视图里显出来;不对外宣称"绝对准确"。

### 5C.4 接口、入口与复用

- FastAPI 增:`GET /papers/{id}/decomposition`(返回拆解视图所需结构化数据,各条带溯源指针)。
- **入口**:在论文库点任一篇 → 进入本视图。
- **复用**:`reading_blocks.json` / `ai_sections.json` / `literature_card.json` /
  `evidence_atoms.json` / `paper_syntheses.json` 既有产物;`ai_client`/`ai_cache` 跑新增的术语表抽取。
- **真理源不变**:各 JSON 为真理源,SQLite 只索引;视图是"组装展示",不改写产物。

## 6. 数据流

### 6.0 发现与下载(攒起步库)

```text
用户搜关键词(OA 源) 或 导入题录(RIS/BibTeX)
  -> 得到带 DOI 的文献清单
  -> 用户勾选
  -> POST /download 起批量下载任务(后台)
  -> 按优先级取 PDF:OA -> (用户启用)Sci-Hub / 截图下载
  -> paper_pool 去重/哈希/题录归一
  -> 落入本地 PDF 库,进入 6.1 抽取
```

### 6.1 导入到可用(一次性,重)

```text
用户选 PDF
  -> POST /papers/import 起抽取任务(后台)
  -> 抽取后端(PyMuPDF 或 Docling)产出 clean package
  -> 引擎跑阅读块/卡片/关联(复用恢复式流水线),写 library/Sxx/*.json
  -> 精细层:证据原子 + 篇内关联 + 术语表(供单篇拆解视图,见 5C)
  -> 存储层 reindex 进 SQLite
  -> 课题组归类(脚本聚类,疑难走 AI)写 groups.json + 索引
  -> 前端论文库/关联网/课题组视图刷新可见
```

### 6.2 写综述(交互,核心)

```text
用户在"写综述"屏发起会话
  -> POST /writing/session 建会话
  -> AI 从关联网/卡片找灵感并抛观点
  -> 用户追问/修正(多轮 POST /writing/turn)
  -> 收敛到关键论点后接地出稿:逐句带可溯源引用,过机械引用/风格门禁
  -> 前端"出稿与溯源"屏展示;点引用 GET /trace 跳回来源原文
  -> 导出
```

### 6.3 请求/任务总体

- 轻请求(浏览、查卡片、看关联网)同步直接返回。
- 重请求(导入抽取、出稿)返回 `job_id`,前端订阅 `GET /jobs/{id}/events` 显示进度,
  失败可续跑。

## 7. 出错处理

- **没填 API key**:进入需要 AI 的步骤前拦截,引导去设置屏;浏览类功能不受影响。
- **抽取后端缺失**:用户选 Docling 但未安装 → 提示并提供按需下载;或回退 PyMuPDF。
- **抽取失败 / 坏 PDF**:沿用现有"只标记不抢救"策略(参见 `config/docling_unresolved.json`),
  在该论文上显式标失败状态,不中断整批。
- **长任务中断**(关窗、断电):任务系统记录分步状态,重启后可续跑,不从头再来。
- **AI 调用失败/限流**:任务标失败 + 可重试;复用 `ai_cache` 避免重复烧钱。
- **SQLite 损坏**:从 JSON 产物重建索引。
- **质量门禁拦截**(机械引用/风格):把拦截原因回传前端,允许按现有规则做一次可追踪的引用格式修复;
  不允许界面绕过门禁直接改 accepted 产物。
- **搜不到 / 取不到全文**(付费墙且未启用兜底):标"仅元数据,无全文",不阻断其余文献。
- **下载插件缺失/不健康**:在来源列表标灰并提示,不影响其他来源与核心功能。
- **Sci-Hub 镜像失效**:提示用户在设置里更换镜像;不内置兜底镜像。

## 8. 安全性

- **API key**:存系统钥匙串;不打进安装包、不写明文配置、不进日志;仅后端进程内使用。
- **本地服务**:FastAPI 仅绑 `127.0.0.1`,不对外暴露;可加进程内 token 防本机其他程序乱调。
- **版权/合规**:做法 A 下,PDF 与抽取结果**只存用户本地**,开发者不接收、不代存,
  把版权风险留在用户侧(这是选做法 A 的主要理由之一)。
- **更新与依赖**:Docling 按需下载需校验来源完整性(哈希/签名),防止下载被篡改。
- **下载层合规姿态**:核心只走开放获取;**Sci-Hub 插件默认关闭、不内置镜像、由用户自启自配自担**;
  截图下载只用用户自己已登录、有权限的会话,不绕过任何登录/验证码/门禁。安装包本身不携带灰色来源。
- **OA 源礼貌访问**:对 OpenAlex/Crossref 等遵守速率限制并带 `mailto`/User-Agent 标识,避免被封。

## 9. 测试策略

- **单元测试**:每个新增组件(抽取接口、任务系统、存储索引、密钥管理)独立可测。
- **契约测试(关键)**:PyMuPDF 与 Docling 两实现都要通过同一份"抽取接口契约"用例,
  保证下游流水线对两者都能跑通;质量差异单独度量并登记 `ISSUES.md`,不断言等价。
- **端到端冒烟**:用 1~2 篇小样 PDF,跑"导入 → 抽取 → 卡片 → 出稿"全链路一次。
- **引擎回归**:沿用现有 `validate_*` 校验脚本,确保包装层没破坏引擎产物。
- **插件契约测试(关键)**:每个下载来源插件都要过同一份 `SourcePlugin` 契约用例(能力声明、
  search/fetch 行为、失败隔离);新加来源也必须先过契约才算合格。
- **发现/下载冒烟**:用 arXiv 小样跑一次"搜索 → 勾选 → 下载 → 去重 → 入抽取"全链路。
- **课题组聚类**:固定一份元数据样本,脚本聚类结果**确定可复现**;AI 裁决用例验证缓存命中后不再重算。
- **单篇拆解视图**:沿用 `validate_evidence_atoms` / `validate_paper_syntheses`,新增 `validate_glossary`;
  另加**溯源完整性测试**——视图里每条结果/关联/术语都必须能解析回某个 `reading_block`。

## 10. 风险与待实现阶段定的决策

- **抽取质量差异**:PyMuPDF 默认会降低抽取质量,可能影响下游卡片/出稿;需契约测试 + 明确告知用户。
- **跨平台打包**:Windows 与 macOS 各需一条打包链;**macOS 代码签名/公证(notarization)** 是
  绕不过的工程成本,放实现计划处理。
- **Docling 按需下载**:数 GB 下载的来源、断点续传、完整性校验的具体机制,实现阶段定。
- **pywebview vs Tauri**:本文件默认 pywebview;若打包/体验不达预期,再评估换 Tauri。
- **空库首启体验**:无样板库时,新用户第一屏的引导(怎么填 key、怎么搜/导入第一篇)需重点打磨,
  否则试用易在门口流失。
- **重做法插件框架成本**:接口稳定性、加载/隔离、版本兼容是额外设计成本;实现阶段先冻结接口 v1。
- **OA 源 API 变动/限流**:OpenAlex/Crossref/arXiv 接口或配额可能变;需可配置、可降级到纯题录导入。
- **Sci-Hub 不稳定与法律风险**:镜像常失效、且法律风险归用户;开发者不维护镜像、不提供默认地址。
- **课题组归类的固有不确定**:学科署名惯例不同(末位≠PI)、常见姓名重名碰撞(尤以缺 ID 的 RIS 导入最弱)、
  PI 跨机构流动会按单位错分。对策:锚点规则可配、ID 信号优先、归并带证据可人工纠错;不对外宣称"绝对准确"。
- **精细层回归的成本与质量**:重新纳入证据原子/篇内关联会增加每篇抽取耗时;其准确度有已知风险
  (I1、约 7% 原子失败),术语表为新抽取需单独评估。对策:点击溯源 + 置信度展示让用户自核,不宣称满分。

## 11. 三阶段如何复用本骨架

- **第二阶段(云端)**:FastAPI 服务从 localhost 搬到服务器;SQLite 换正式数据库 + 对象存储;
  新增账号/权限/共享缓存;前端从"连本地"改为"连服务器",界面代码大体复用。
- **第三阶段(手机)**:同一套 FastAPI 后端,前端因是网页技术,可复用为移动端/小程序的视图层。

> 本阶段的每个边界(网页前端、HTTP 接口、可插拔抽取、JSON 为真理源)都是为后两阶段省返工而设。

## 12. v1 里程碑(建议顺序,细化留给实现计划)

1. **M1 骨架打通**:FastAPI 起服务 + pywebview 起窗口 + 一个最小接口("查空库")跑通。
2. **M2 抽取插槽**:统一抽取接口 + PyMuPDF 实现 + 契约测试;导入一篇 PDF 跑到卡片。
3. **M3 发现+下载层**:`SourcePlugin` 接口 v1 冻结 + OA 内置源(搜索/导入题录/批量下载)
   + 复用 paper_pool 去重;Sci-Hub / 截图下载插件作为参考实现随后补。
4. **M4 浏览**:存储索引 + 论文库屏 + 关联网屏 + 课题组归类(脚本主路 A + 辅路 C;AI 兜疑难随后补)。
5. **M5 单篇拆解视图**:精细层回归(证据原子 + 篇内关联)+ 新增术语表抽取 + 点击溯源阅读页。
6. **M6 核心**:找灵感 + 追问式写综述 + 出稿与溯源(接通门禁)。
7. **M7 打磨与打包**:设置/密钥、Docling 按需安装、Windows/macOS 打包与签名、空库引导。
