- 日期:2026-06-09 初稿;**2026-06-10 升 v2**(并入用户旅程校准后的新增:核心度/单篇特写/时间与机构镜头/图墙/阅读路线/检索屏联动计数)
- 状态:各项已与用户逐项确认(对话内);**本文件待用户整体审阅**;实施排在「管线换代」之后
- 单一事实源:数据 schema 见 `2026-06-10-data-framework.md`;管线侧前置见 `../archive/2026-06-10-pipeline-regeneration-design.md`(finding 类、authorship、机构注册表)
- 缘起:用户对现状的判断——「目前的东西像一个工具库,不太像一个给人使用的好东西」

## 0. 产品观(用户原话的翻译)

库不是仓库,是**一张会生长的地图**;打开 app 即见地图。两个核心画面:
1. **知识地图**:每篇论文一个点,相近的聚成有名字的"区"(分区即导航);
2. **新文献着陆**:新导入的论文降落在地图上,高亮并连线到最亲的老文献——
   "它和我已有的什么有关"一眼可见;落在空白地带 = 显式信号(惊喜或误入库)。

## 1. 布局(已选定:A 为主 + B 的滑出卡)

- **全屏地图即首页**:地图占满视口;其余全部是悬浮件——
  顶左:logo + **镜头切换**(主题 ▾);顶中:大搜索框(要素/论文,即现有 /elements 检索);
  顶右:状态角标(N 篇 · 要素覆盖 k/N · 后台任务进度);
  右下:「m 篇新文献着陆 →」角标。
- **新着陆滑出卡**(借 B):点右下角标,从右侧滑出面板列出每篇新文献:
  `S352 → 落入模拟区(最近邻 S218/S341)` / `S354 ⚠ 空白地带(与现有库关联弱)`;
  点条目地图飞行定位。平时收起,不挡地图。
- **左侧细轨导航**:找 / 读 / 写 / 设置(见 §5)。

## 2. 镜头(可切换分区方案;用户拍板"必须可切换")

| 镜头 | 分区依据 | 数据来源 | 可用时点 |
|---|---|---|---|
| 主题(默认) | topic 稳定 ID(经注册表归一的领域标签) | 卡片 domain_tags → topic 注册表 | **261 篇即刻全亮** |
| 方法 | preparation/measurement/simulation 类要素 | 要素索引 | 随索引覆盖逐篇点亮 |
| 材料 | material 类要素 | 要素索引 | 随索引覆盖逐篇点亮 |
| 时间(v2 新) | 年代分层染色 + 时间轴拖动 | 卡片 `paper.year` | 即刻全亮 |
| 机构(v2 新) | 机构注册表稳定 ID;机构内论文数 >1 才展开**团队**(资深作者)下钻;再上一层国家/地区粗分 | `authorship.json` + `data/institutions/` | 随 authorship 拉取点亮 |

- **核心度(v2 新)**:所有镜头下,**点的大小 = 共享要素加权连接度**——"先看和别人关联最紧密的核心文章"
  一眼可得;新手入口。
- **机构 × 要素交叉(v2 新)**:机构卡片显示该机构论文的要素分布("这个所偏 GCMC / 那个组全是实验"),
  数据 = 要素索引按机构过滤,零新调用。
- **要素首现(v2 新)**:时间镜头附带"某要素在本库最早出现于哪年哪篇"(elements × year 派生,机械)。

- **渐进点亮**:没有要素的论文在方法/材料镜头下显示为灰点(可点,提示"该篇要素未构建");
  这把"构建进度"变成看得见的产品体验,也不阻塞首发(主题镜头第一天就完整)。
- 镜头枚举不写死:从注册表 facets 读;将来新增 facet(含 `proposed:` 转正的)自动成为候选镜头。

## 3. 分区与布点算法

- **相似度**:两篇共享要素的 **IDF 加权和**(罕见要素加分大,"都研究吸附"几乎不加分)——
  复用候选边的已验证配方(`build_candidate_edges.py` 的思想),但算在**要素**上,且按当前镜头
  只取该镜头的 facet 子集;主题镜头退化为 domain_tags 的同款加权。
- **聚类→分区**:相似度图上做社区发现(沿用 graph.js 已有的 Louvain 思路);
  每区命名 = 区内**最高频且 IDF 不过低**的 1-2 个要素/标签(机械规则,无 AI)。
- **布点**:力导向 + 同区聚拢(cluster gravity);**位置要稳定**:
  - 布局结果缓存(`data/elements/map_layout_<lens>.json`);
  - 新论文**增量放入**现有地图(以最近邻为锚初始化,只局部弛豫),老点不重排;
  - 全量重排只在两种情况:用户显式点"重新布局",或镜头切换首次计算。
  - 缓存文件 = 可重生产物(删了重算即可),不属长期状态。
- 老关系网(supports/contradicts 边)**不参与布点**,将来作为可叠加图层画在地图上(范围外,见 §7)。

## 4. 交互

- **点区**:放大该区 + 右侧列出成员论文(按要素数排序)。
- **点单篇点**:右侧浮出论文卡:标题/年份/期刊 + 该篇要素清单(按 facet 分组,带引文预览)+
  「进拆解页」「在检索屏中选中该篇要素」「**关系特写**」三个出口。
- **单篇关系特写(v2 新)**:自我网络视图——该篇居中,邻居按关系分色环绕
  (支持/矛盾/互补 = AI 判边;共享要素强度 = 机械边),点邻居可跳转;
  这是"看这篇文章和其他文章关系网的特写"的直接实现,也是旧关系网图谱的新家。
- **新着陆**:定义 = **最近一次导入批次**(import job 完成时间最新的一组);红点 + 呼吸光圈,
  虚线连到 top-3 最近邻老文献;空白地带(与任何区的相似度低于阈值)给 ⚠ 标记和文案。
- **搜索框**:命中要素 → 地图上含该要素的点高亮(其余压暗),等于"在地图上做检索";
  命中论文 → 飞行定位。
- 悬停:点 → 标题 tooltip;区名 → 该区规模与 top 要素。

## 4b. 配套视图(v2 新)

- **图表墙**:按论文/按区浏览 Docling 已抽取的全部图片缩略图(`library/<id>/figures/`,数据已在、
  从未展示);点开大图 + 所在段落锚点。入口:「读」组 + 论文卡。零 AI。
- **阅读路线**:对当前区/当前选择生成推荐阅读顺序(综述先行 → 核心度降序 → 外围),
  纯机械(paper_type + 核心度排序);文案如"建议从这 3 篇入手"。入口:地图区卡片 + 「读」组。

## 4c. 检索屏 v1.1(v2 新;分面联动)

- **联动计数**:选中任一要素后,左树全量重算"与已选组合的交集计数"
  (`POST /elements/refine`:一次返回命中论文 + 全树条件计数);零结果项灰显沉底。
- **热度排序 + 折叠**:每 facet 默认展开联动计数 top-10,其余收入"展开全部";树顶加树内搜索框。
- **长尾分层**:仅 1 篇命中的条目默认收进"其余 N 项(各 1 篇)"折叠组——细粒度是检索资产,
  浏览靠分层,不靠砍分类。
- 共现抽屉与联动计数同源(同一查询族),后端复用。

## 5. 导航重组(收敛 8 项 → 4 组)

左轨:**找**(要素检索 + 全库统计)/ **读**(论文列表 + 单篇拆解)/ **写**(写作)/ **设置**
(设置 + 研究组并入)。地图 = 首页(点 logo 回地图)。「关系网」从导航撤下,
其功能并入地图图层(范围外)+ 保留旧路由可直达(不删代码)。
旧 hash 路由全部保留(书签兼容);只改 nav 的呈现与分组。

## 6. 数据与 API(草案)

- 新桌面服务模块 `map/`:从 elements_index.sqlite + 卡片标签计算相似度/聚类/布局,
  产物缓存进 `data/elements/map_layout_<lens>.json`(含 schema_version、lens、节点坐标、
  分区成员、区名、参数指纹;参数变了自动重算)。
- 端点(草案):`GET /map?lens=<lens-id>`(布局+分区+点亮状态;首发三个 lens,枚举随注册表 facets 扩展)、
  `GET /map/arrivals`(最近批次 + 最近邻)、`POST /map/relayout`(显式全量重排)。
- 计算量级:261 篇 × 要素集合交集,毫秒~秒级,后台任务可选;不调 AI,零成本。

## 7. 范围外(本 SP 不做,记录防爬行)

- 全局关系边叠加图层(单篇特写已覆盖主要需求;全局叠加后看)。
- 框选多篇 → 送写作素材/对比表——配合 SP5。
- 引用关系镜头(需滚雪球数据)、库内复现度(需 finding 级关系,SP3 后)。
- 个人层(已读/星标/批注)= 独立小 SP,排在裁决台账旁,不混入本期。
- 视觉品牌化(配色/字体/署名)单独一轮,不混在信息架构里。

## 8. 实施排序(v2 更新)

1. 全库要素构建收尾与验收(进行中)→ **管线换代**(SP-Regen:finding/卡片v3/topic与机构注册表/并行化——本 SP 的多个镜头依赖它);
