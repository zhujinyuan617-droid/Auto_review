# 研究要素索引 + 检索/统计屏(SP1+SP2)设计

- 日期:2026-06-09
- 状态:设计已与用户逐节确认(对话内);**未实施,未提交**
- 上游讨论:本轮头脑风暴(科研痛点 → 产品路线 SP1–SP6 → 选定 SP1+SP2 先行)
- 下游:批准后由 writing-plans 产出实施计划

## 0. 背景与动机

现有整理方式是「粗标签(topic/method/object)+ 字面匹配」,与研究者实际提问方式不符。
三个代表性问题,现在都答不了:

1. 「我实验室只有球磨机 → 所有用过球磨的文章」——按方法/设备找全(球磨往往只出现在
   实验段落,不在标签;ball milling / mechanical milling 写法一变即漏)。
2. 「大部分文章用什么表征?」——全库分布统计,无此视图。
3. 「他们具体怎么做的分析?」——从统计钻到某篇的逐字原文,无此通路。

用户定位:**长远当产品做**(面向更广科研用户)。本子项目是产品路线的第一块地基:

| 子项目 | 内容 | 本期? |
|---|---|---|
| SP1 | 研究要素索引(引擎) | **是** |
| SP2 | 检索/统计屏(桌面) | **是** |
| SP3 | 裁决台账 + 查证台 | 否(注册表人工操作日志为其种子) |
| SP4 | 增量更新 + diff 报告 | 否(流式注册表天然增量,部分化解) |
| SP5 | 写作交互(门禁当检查器、成品对比表) | 否 |
| SP6 | 产品化硬底(打包/多库/引导) | 否(语言闸提前到本期) |

## 1. 目标与非目标

**目标**
- 上述三连问在桌面 app 内点几下可答,答案钻到逐字原文。
- 全自动:存量一键引导、增量随导入自动长,全程无需人工干预(P0)。
- 每条要素可溯源(逐字锚点,机械核真),AI 结果不核真不入库。
- 要素 ID 稳定,人工修正永久保存,扩容/重抽不丢。

**非目标(本期不做,防范围爬行)**
- 跳转 PDF 原页(先跳 block 段落视图;PDF 页锚点留待后续)。
- 成品对比表生成器(检索屏表格模式覆盖约七成需求;带数值成品表归 SP5)。
- 裁决台账与查证台全量(SP3)、滚雪球检索、旧图谱/旧词表改造。
- embedding(语义向量)匹配层:本期只留接口,不实现(理由见 §4 流式期)。

## 2. 设计原则

- **P0 默认全自动,人工只是可选增益**:
  (a) 流程上——导入→抽取→归一→索引一条龙后台跑完,失败自动重试、坏文件自动隔离,
  用户永不跑脚本、永不改 JSON;
  (b) 数据上——AI 自动产出直接可用(锚点核真保信任下限),人工改名/合并是可选精修,
  做了更准且永久保存,不做不挡任何功能。
- **锚点核真一票否决**:AI 报的每条要素,引文必须逐字回到 block 验证(容忍空格/标点/
  省略号差异,参照 `scripts/audit/audit_card_grounding.py` 与 `scripts/use/verify_angle.py`
  已踩稳的容错匹配做法);核不过即丢弃并计数。宁可漏,不可编。
- **AI 提议、人终裁**:沿用架构 v2 哲学。AI 的归并/匹配都是提议;人的修改是最终的,
  写入追加式日志,重建自动接回。
- **纯新增层**:不动 blocks、不动卡片、不动边。现有图谱降级为辅助视图,本期不碰。
- **「用了」不等于「提到了」**:每条要素带角色标签 `used`(本篇自己做的)/
  `mentioned`(仅引述他人,含综述文章的情形)。统计默认只计 `used`,否则约 13 篇
  综述文章会搅浑全库分布。

## 3. 架构与数据流

```text
library/Sxx/ blocks(逐字原文,已有,不动)
   │  ① AI 要素抽取(每篇一次;读 blocks)
   ▼
library/Sxx/elements.json(本篇全部要素出现;每条带逐字锚点,核真后入库)
   │  ② 注册表匹配/归一(到货匹配;详见 §4)
   ▼
data/elements/registry.json + registry_log.jsonl(规范名+同义词族+稳定 ID;只生不改)
   │  ③ 建索引(纯脚本,可随时从 ①② 重建)
   ▼
data/elements/elements_index.sqlite(要素→论文→锚点;计数/分组/组合过滤)
   │  ④ desktop API(本地 FastAPI,见 §7)
   ▼
屏A 检索(三栏工作台)        屏B 统计(总览→大图→共现)
```

职责分工沿用现行模式:抽取/归一/索引逻辑在**引擎**(`Document_Decomposer`,新模块 +
`scripts/elements/`);桌面 app 经 `engine_bridge` 包装为后台任务与界面,不改引擎已有代码。

## 4. 分类机制:种子 → 引导 → 流式(三段式)

用户决策:分类不由开发者拍死,由 AI 从语料中得出;开发者只给「预备内容」。
落地为三段式,同时避开两个已知坑(I12 批量归一非确定/过度合并;先到先得路径依赖):

**① 种子(`config/element_seeds.json`,进 git,随产品分发)**
- 内容:大类骨架(制备/处理、测量方法、表征、模拟/计算、分析内容、材料/对象、条件
  ——作为初值)+ 常见同义词族(如 XRD = X-ray diffraction;GCMC = grand canonical
  Monte Carlo)。
- 地位:**先验,不是法律**。引导期可拆类、并类、加新类、改名。

**② 引导期(一次性批任务,对现有全库)**
- 全库开放式抽取(不限于种子词)→ 核真 → **一次**全局归并:AI 看全部抽得的要素,
  决定本库实际需要的类目与要素表,产出注册表 v1。
- 防过度合并纪律(I12 教训):禁大杂烩桶、单次合并组容量上限、归并后机械审计超大桶
  (成员数异常的条目标记复查)。
- 非确定性被「只跑一次」冻结:此后不再全量重归并,漂移风险不随扩容复发。

**③ 流式期(此后每篇新论文)**
- 到货匹配:每条新要素 → 精确/别名匹配 → 不中则 AI 对照同大类注册表候选判定
  (注册表量级为数百条,候选可全量入提示词)→ 仍不中则**开新条目,绝不强塞**。
- 新大类可由 AI 提议(预期罕见,打标记待复查)。
- embedding 层:本期不做,在匹配器处留接口(`Matcher` 可插拔);注册表规模增大后再加。
- 白送的两个性质:**天然增量**(新论文只是追加,无全库重建)、**ID 天然稳定**
  (条目只生不改)。
- 路径依赖缓解:展示名(display name)可随使用频次自动切换为最常见写法,ID 不变。
- 合并审计(周期性、建议制):AI 提议疑似重复对;自动应用走**重定向**(redirect_to,
  两个 ID 都保留),不删不丢,可回退。

## 5. 数据模型

**要素出现(occurrence,存于 `library/Sxx/elements.json`)**

```json
{
  "facet": "preparation",
  "surface": "ball-milled for 4 h",
  "canonical_id": "elem:preparation/ball-milling",
  "quote": "The montmorillonite was ball-milled for 4 h at 400 rpm under N2 ...",
  "reading_block_id": "S123-RB-0047",
  "role": "used",
  "value": {"duration": "4 h", "speed": "400 rpm"},
  "quote_verified": true
}
```

> 更正:初稿此例误写锚点为 `"block_id": "S123#b047"`;引擎真实锚点是 reading_blocks.json
> 的 `reading_block_id`(格式 `Sxx-RB-NNNN`,如 `S09-RB-0003`),已按实改正。

- `value` 仅条件类要素有,从已核真的 quote 用机械规则解析数值/单位;解析失败只留原文。
- `quote_verified=false` 的记录不写入(丢弃前计数,进任务报告)。

**注册表条目(`data/elements/registry.json`)**

```json
{
  "id": "elem:preparation/ball-milling",
  "facet": "preparation",
  "display_name": "ball milling",
  "aliases": ["mechanical milling", "ball-milled", "球磨"],
  "redirect_to": null,
  "origin": "bootstrap",
  "human_locked": false
}
```

- ID 规则:`elem:<facet>/<slug>`,首次分配后**永不改、永不重用**;合并=设 `redirect_to`。
- facet 与要素同理:内部用英文 slug(如 `preparation`)作稳定 ID,界面显示中文名
  (如「制备/处理」);两者映射在注册表里,改显示名不动 slug。
- `human_locked=true` 表示人工定过名/归属,自动流程不得再动(合并审计仍可提议)。

**流水账(`data/elements/registry_log.jsonl`,追加式)**
- 每行一个事件:创建/收别名/合并重定向/人工改名/人工合并/审计建议,带时间与来源
  (auto-bootstrap / auto-stream / audit / human)。
- 这是 SP3 裁决台账的种子:人工事件即裁决,重建时按日志重放接回。

**存放与重建关系**
- `library/Sxx/elements.json`:可随时对单篇重抽重生(幂等;ID 与人工修改都在注册表,不受影响)。
- `data/elements/`:**长期状态**,不可当报告随意删除;SQLite 索引随时可从 elements.json
  + registry 重建。
- 种子在 git;以上其余均不进 git(沿现行 gitignore 约定)。

## 6. 管线与自动化(P0 落地)

**存量:引导任务(桌面端一键,后台 job)**
- 步骤:并行抽取全库(沿用现有并行/重试模式)→ 核真 → 全局归并 → 建注册表 → 建索引。
- 量级估计(按既往经验 [估计,非实测]:DeepSeek 约 0.2 元/篇、抽取为单次调用):全库
  数百元以内、数小时后台跑完;界面显示进度与失败清单。
- 跑完即用;抽样审计(§9)另行安排,不阻塞使用。

**增量:挂进现有导入流程**
- 桌面导入 job 在 sections→reading→card 之后追加 elements 阶段(抽取+匹配+索引更新),
  单篇约多 1 分钟;引擎侧给对应 CLI 脚本(开发/批处理用,用户不接触)。

**语言闸(根治 ISSUES I17 的新导入路径)**
- 导入流程最前端检测正文 CJK 占比,超阈值 → 自动隔离并界面提示「中文文献暂不支持,
  已搁置」,不进 AI 阶段、不崩溃。

**失败处理**
- AI 阶段失败:自动重试;仍失败标记该篇「要素待补」,不挡其余功能;界面显示覆盖率
  (如「要素层 258/261」),可一键重试待补集。

## 7. API 面(desktop,本地)

| Endpoint | 用途 |
|---|---|
| `GET /elements/overview` | 屏B 落地页:各 facet top-N 小图数据 |
| `GET /elements/stats?facet=` | 屏B 大图:该 facet 全量计数(默认 role=used) |
| `GET /elements?q=&facet=` | 检索/自动补全(命中规范名+别名;预留 semantic 参数) |
| `GET /elements/{id}` | 条目详情:论文列表 + 各篇命中 quote + 锚点 |
| `GET /elements/{id}/cooccurrence` | 共现面板:同篇共现要素按 facet 分组计数 |
| `POST /elements/query` | 组合查询(多要素 AND/facet 过滤)→ 论文+命中明细;驱动列表与表格模式 |
| `GET /papers/{id}/elements` | 单篇要素清单(并入现有论文详情页) |
| `PUT /elements/{id}` | 人工改名/加别名/合并(写 registry_log,置 human_locked) |
| 任务 | 复用现有 `POST` 任务 + `GET /jobs/{id}` 模式(引导任务、待补重试) |

## 8. 界面设计(已与用户在可视化 mockup 上确认)

**屏A 检索 = 三栏工作台 + 搜索引擎顶栏 + 表格切换**(A 为主体,融 B、C)
- 顶部:大搜索框,自动补全提示同义词归并(「球磨 = ball milling = mechanical milling」),
  支持多要素 chip 组合(AND)。
- 左栏:facet 分类树(各项带计数),即浏览模式,勾选即过滤。
- 中栏:命中论文列表,**列表 ⇄ 表格**切换;表格模式列出共同要素与条件参数列,可导出 CSV。
- 右栏:选中论文的命中要素 + 高亮逐字引文 + 「跳到原文段」(block 视图)。

**屏B 统计 = 分层导航:总览 → 大图 → 共现**
- 落地页(总览仪表盘):每个 facet 一张 top-5 小图,看库的全貌;点小图进大图。
- 大图:该 facet 排序条形图(显示分母,如 80/261);点条目 → 右侧抽屉:论文列表 + 引文。
- 抽屉内「配套要素」面板(共现):锚定该要素后,同篇共现的其他要素按 facet 分组
  (如「球磨 12 篇里:XRD 10/12、BET 9/12」),点击可继续钻取。
- 统计默认 role=used;可显式切换包含 mentioned。

## 9. 质量与验证(按仓库家规:证据+方法+范围)

- **机械层**:occurrence 100% 引文核真(结构保证,核不过不入库)。
- **抽样层**:引导期跑完后抽 20 篇,人/AI 对照原文方法/实验章节,统计漏报与误报,
  结果记入 `ISSUES.md` 新条目;**抽样完成前,任何文档不得写「已验证/合格」**。
- **测试**:引擎侧单测覆盖确定性部分(核真匹配、值解析、注册表操作、重定向、日志重放);
  桌面侧沿用 fake-AI 模式测 API 与任务;一篇金样本(固定假 AI 响应)端到端。
- **已知风险(诚实预登记)**:
  (a) 抽取漏报——方法藏在表格/图注/长文深处(对策:抽样量化,不预claim 召回率);
  (b) 引导期归并漂移——单次冻结但那一次仍可能并错(对策:超大桶审计 + 人工可改且永久);
  (c) 流式匹配错挂——AI 把新要素挂到错误条目(对策:合并审计建议制 + 重定向可回退)。
  三者均按「线索可修正」设计,不当真值。
